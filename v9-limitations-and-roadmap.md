# v9 DB의 한계 및 개선 로드맵

**작성일**: 2026-04-04
**주제**: 메모리 전용 시스템의 DB 분류 타당성
**결론**: 파일 저장 기능이 필수적

---

## ⚠️ v9 DB의 핵심 문제

### 1. "저장 안 되는데 DB라고 불러도 되나?"

**정직한 답**: ❌ **현재로서는 불완전한 DB이다**

#### 메모리 DB의 정의
```
✅ 데이터 구조: 있음 (테이블, 인덱스)
✅ 쿼리 언어: 있음 (SQL)
✅ 트랜잭션: 있음 (ACID)
❌ 영속성: 없음 (메모리만)

결론: "데이터베이스 엔진"이지 "데이터베이스"는 아님
```

### 2. 현실의 문제점

```
시나리오                     v9 DB의 문제
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
프로그램 종료          → 모든 데이터 손실 ❌
서버 재시작            → 모든 데이터 손실 ❌
전원 차단              → 모든 데이터 손실 ❌
네트워크 끊김          → 모든 데이터 손실 ❌
대용량 데이터          → 메모리 부족 ❌
```

### 3. 진정한 "DB"의 조건

| 조건 | 설명 | v9 DB |
|------|------|-------|
| **데이터 구조** | 테이블, 인덱스 등 | ✅ |
| **쿼리 언어** | SQL 또는 유사 | ✅ |
| **트랜잭션** | ACID 보장 | ✅ |
| **영속성** ⭐ | 디스크에 저장 | ❌ |
| **복구 가능성** | 재시작 후 복원 | ❌ |
| **동시성** | 다중 접속 | ⚠️ |

## 📊 정직한 분류

### 4. v9 DB의 올바른 이름

```
❌ 잘못된 이름
├─ "데이터베이스" (DB)
├─ "영속 저장소"
└─ "프로덕션 데이터베이스"

✅ 올바른 이름
├─ "메모리 DB 엔진"
├─ "인메모리 데이터 구조"
├─ "쿼리 가능한 캐시"
└─ "데이터베이스 프로토타입"
```

### 5. 기술적 분류

```
데이터베이스 분류
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. 영속 데이터베이스
   ├─ PostgreSQL (관계형 + 영속)
   ├─ SQLite (관계형 + 파일)
   └─ MongoDB (문서 + 영속)

2. 인메모리 데이터베이스
   ├─ Redis (KV 캐시)
   ├─ Memcached (캐시)
   └─ v9 DB ← 이 위치 (인메모리 + SQL)

3. 데이터베이스 엔진 (라이브러리)
   ├─ v9 DB-core
   └─ SQLite (임베디드 모드)
```

## ✅ 해결책: 파일 저장 기능 추가

### 8. v3.0에서 추가해야 할 기능

#### 8.1 파일 저장 (필수)
```freelang
// 데이터베이스를 파일로 저장
var db_data = serialize_database(database)
file_write("database.v9db", db_data)

// 파일에서 복원
var loaded_data = file_read("database.v9db")
var restored_db = deserialize_database(loaded_data)
```

#### 8.2 자동 체크포인트 (필수)
```
주기            사항
━━━━━━━━━━━━━━━━━━━━━━━━
10초마다        메모리 → 디스크 쓰기
1분마다         변경사항 저장
정상 종료시     전체 저장
```

#### 8.3 WAL (Write-Ahead Logging, 중요)
```
개선 전: UPDATE 실패 → 데이터 손상 위험
개선 후: 모든 변경사항 미리 로그에 기록 → 복구 가능
```

## 📈 개선 로드맵 (수정된)

### 9. v2.0 → v3.0 → v4.0 진화

```
v2.0 (현재)
├─ 기능: 메모리 DB 엔진
├─ 영속성: ❌ 없음
├─ 적합도: 📚 교육용만
└─ 평가: "DB가 아닌 엔진"

       ↓ (v3.0 = 진정한 DB)

v3.0 (필수)
├─ 기능: + 파일 저장
├─ 기능: + WAL 로깅
├─ 기능: + 자동 체크포인트
├─ 영속성: ✅ 있음
├─ 적합도: 🎮 게임, 🧪 프로토타입
└─ 평가: "완전한 경량 DB"

       ↓ (v4.0 = 기능 확대)

v4.0 (권장)
├─ 기능: + 다중 스레드
├─ 기능: + 네트워크 서버
├─ 기능: + 복제 (기초)
├─ 적합도: 🏢 소규모 서비스
└─ 평가: "SQLite 경쟁 수준"

       ↓ (v5.0+ = 도전)

v5.0+ (목표)
├─ 기능: + 클러스터링
├─ 기능: + 분산 트랜잭션
├─ 적합도: 📊 중규모 서비스
└─ 평가: "PostgreSQL 방향 추진"
```

## 📋 개선 계획 (현실적)

### 12. v3.0 로드맵 (필수 기능)

#### Phase 1: 파일 I/O (1주)
```
└─ serialize_database()
└─ deserialize_database()
└─ file_write() 통합
└─ file_read() 통합
```

#### Phase 2: WAL 로깅 (1주)
```
└─ 모든 변경사항 로그
└─ 트랜잭션 로그
└─ 자동 복구
```

#### Phase 3: 체크포인트 (1주)
```
└─ 자동 저장 (10초)
└─ 수동 저장 (COMMIT)
└─ 정상 종료 저장
```

#### Phase 4: 테스트 (1주)
```
└─ 파일 무결성 검증
└─ 복구 테스트
└─ 성능 벤치마크
```
